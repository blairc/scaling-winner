Proposal: AI Assistant for Basketball GM

Build an optional, low-friction assistant layer that helps with roster decisions, lineup suggestions, and long-term planning while keeping the "you are still the GM" spirit of Basketball GM.

1. Product Vision

The assistant should reduce repetitive work without removing player agency. Instead of auto-playing the game, it should provide concise recommendations, confidence scores, and one-click actions for common workflows.

Design Principle: Advisory-First

Default behavior is suggestion + rationale. Full automation can be explicitly enabled per workflow.

Design Principle: Context-Aware

Recommendations adapt to team state (contending vs rebuilding), contract windows, injuries, and league economics.

Design Principle: Reversible

Every assistant action should be previewable and easy to undo from a session log.

2. Core Use Cases (Basketball GM First)

Roster + Rotation Triage

  • Who should start/bench based on current roster and chemistry goals.
  • Minute allocation suggestions with fatigue and development weighting.
  • "What changed?" explanations after major injuries or trades.

Trade Assistant

  • Generate realistic trade targets for selected strategy (win-now/rebuild).
  • Show expected value change: short-term wins, future assets, payroll impact.
  • Rank candidate offers and provide a risk breakdown.

Contract + Cap Management

  • Highlight extension candidates and likely negotiation outcomes.
  • Warn about future cap crunches and tax risk timelines.
  • Suggest cap-preserving alternatives to expensive moves.

Draft Support

  • Draft board recommendations by team timeline and roster needs.
  • Trade-up/trade-down prompts with expected value delta.
  • Post-draft development plan template for rookies.

3. Technical Architecture Options

Option A: Client-Side Userscript (Fastest)

Build as a Tampermonkey/Greasemonkey script injected into basketball-gm.com. Read in-game state from existing data stores and overlay a lightweight assistant panel.

Pros: quick to prototype, private by default, no backend required.

Cons: brittle against upstream UI/internal API changes.

Option B: Browser Extension (Recommended)

Package assistant as an extension with content script + isolated UI. Enables better permissions, settings sync, and controlled updates.

Pros: robust UX, easier distribution/versioning.

Cons: higher initial engineering overhead.

Option C: Optional Companion Service

Add a small backend for heavy analysis and model orchestration when users opt in.

Pros: stronger modeling, cross-save analytics.

Cons: privacy + hosting complexity.

4. AI Strategy

Use a hybrid approach rather than pure LLM decision-making:

  • Deterministic engine: cap math, roster constraints, legality checks, and simulation-based scoring.
  • LLM layer: convert structured analysis into short, readable recommendations and answer "why" questions.
  • Policy layer: user-defined style sliders (aggressive, balanced, conservative) that modulate recommendations.

5. Phased Delivery Plan

Phase 0 (1-2 weeks): Feasibility Spike

Build a local userscript proof-of-concept that reads roster + payroll, then suggests a depth chart with explanation text.

Phase 1 (2-4 weeks): MVP Assistant Panel

Add lineup recommendations, basic trade target ranking, and action history.

Phase 2 (4-8 weeks): Strategic Features

Add contract forecasting, draft board support, and scenario simulation.

Phase 3: Harden + Extend to Other Zen GM Games

Extract a shared core and adapt sport-specific modules for Football GM, Baseball, and Hockey variants.

6. Key Risks and Mitigations

  • Upstream game updates break integrations: isolate data access behind adapters and add regression tests.
  • Over-automation hurts fun: keep recommendations concise, optional, and user-tunable.
  • Trust gap: always show rationale + confidence + fallback alternatives.

7. Immediate Next Steps

  1. Choose delivery mode: userscript (speed) vs extension (durability).
  2. Define data contract from Basketball GM state to assistant engine.
  3. Implement first workflow: roster/rotation recommendation.
  4. Run with 3-5 test franchises (contender, mid-tier, rebuild, cap-strained).
This proposal is intentionally Basketball GM-first, with architecture that can later generalize across the broader Zen GM ecosystem.
